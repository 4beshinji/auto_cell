# auto_cell — iPSC 浮遊バイオリアクター AI/ML 制御システム

auto_cell は、ヒト iPS 細胞の 3D 浮遊バイオリアクター培養を自動化するための制御ソフトウェアフレームワークである。温度、pH、溶存酸素（DO）、生細胞密度（VCD）、グルコース、乳酸、凝集体径などのプロセスデータをリアルタイムに取得し、決定的制御ルール、機械学習、および大規模言語モデル（LLM）を組み合わせた階層型アーキテクチャにより、給餌、灌流、撹拌、ガス制御、継代などの操作を判断・実行する。実装置への接続に先立ち、文献に基づく数理モデル（デジタルツイン）を用いて制御ロジックの検証を行うことができる。

---

## 1. 目的と背景

iPS 細胞を医薬品開発や再生医療に応用するためには、大量かつ均一な品質の細胞を安定的に供給する必要がある。攪拌槽型または Vertical-Wheel 型バイオリアクターによる 3D 浮遊凝集体培養はスケールアップに適した手法であるが、高密度培養下では栄養管理、代謝副産物制御、酸素供給、pH 維持、凝集体サイズ管理、継代タイミングなどの判断が複雑化する。

本プロジェクトは、これらの運転判断をソフトウェアに委ねるための制御基盤を構築することを目的とする。細胞培養に関する知見は設計の前提として組み込まれ、実装の重点はセンサデータの解釈、安全な行動決定、監査可能性、および規制対応の実現に置かれている。

現時点では研究・開発段階（R&D）であり、将来の医薬品製造（GMP）への対応を見据えた設計思想を持つ。

| 項目 | 内容 |
|------|------|
| 細胞 | ヒト iPS 細胞（hiPSC） |
| 培養形態 | 3D 浮遊凝集体培養 |

## 3. 技術的アーキテクチャ

### 3.1 階層型制御構成

本システムは、AI/ML の役割を明確に分離した 3 層の制御階層により構成される。

| 層 | 名称 | 役割 | 例 |
|---|---|---|---|
| **L0/L1** | 決定的監督制御 | 検証済みの決定的ルールに基づき、高速かつ安全な動作を実行する | 乳酸閾値超過時の灌流上昇、凝集体径超過時の撹拌変更 |
| **L2** | 機械学習による最適化 | バッチまたはラン単位で培養条件（CPP）を最適化する | 培地組成、灌流レートの offline 最適化 |
| **L3** | LLM オーケストレータ | イベント駆動で複雑な状況判断、説明生成、アラート作成を行う | 異常時の要因推定、HMI への自然文アラート、承認依頼の作成 |

重要な設計思想は、クリティカルな高速制御を LLM に委ねないことである。秒オーダーの pH/DO/温度制御は装置側の PID コントローラに任せ、AI 層は設定点変更、離散的な操作指示、異常時の判断支援など、比較的遅い監督制御に限定する。

### 3.2 Sense-Decide-Act ループ

システムは以下のループにより自律制御を行う。

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  Sense  │ ──▶ │ Decide  │ ──▶ │   Act   │
│  観測   │     │  判断   │     │  実行   │
└─────────┘     └─────────┘     └─────────┘
     ▲                                  │
     └──────────── 状態変化 ────────────┘
```

1. **Sense（観測）**: センサ、画像、at-line 分析器からデータを取得し、内部の WorldModel を更新する。
2. **Decide（判断）**: 現在の状態、目標値、過去の履歴に基づき、次に実行すべき行動を選択する。
3. **Act（実行）**: 選択した行動をアクチュエータ（ポンプ、撹拌、ガス、サンプリング装置）に送信する。

本ループは ReAct（Reasoning + Acting）スタイルで設計されているが、すべての制御周期で LLM が動作するわけではない。重要なイベント発生時または承認が必要な場面で L3-LLM が呼び出される。

### 3.3 Human-in-the-loop

本システムは、重要な判断において人間の意思決定を組み込む設計とする。

- **通常運転**: L0/L1 の決定的ルールに基づき自動実行される。
- **要承認アクション**: 継代、大きな灌流変更、異常時の継続または停止などは、システムが提案し、HMI で研究者が承認した後に実行される。
- **アラートと説明**: LLM が自然言語により判断根拠を提示し、人間が納得した上で意思決定できるようにする。

この設計は、GMP および AI 規制（EU AI Act、Annex 22 など）で求められる説明責任（accountability）に対応するためのものである。

## 4. システム構成

```
auto_cell/
├── src/auto_cell/
│   ├── plugins/cell_culture/    # 細胞培養ドメインの制御知識
│   │   ├── environment.py       # 培養状態のデータモデル（WorldModel 用）
│   │   ├── channels.py          # センサチャネル定義と取り込みルール
│   │   ├── events.py            # 異常・トリガーイベントの判定ロジック
│   │   ├── tools.py             # 副作用を持つ操作の定義
│   │   ├── sanitizer.py         # 安全包絡線・変化率制限
│   │   ├── prompt.py            # LLM への指示プロンプト
│   │   ├── aggregate_imaging.py # at-line 凝集体画像解析（Cellpose + fallback）
│   │   ├── confidence.py        # AI 判断の信頼度スコア
│   │   └── llm_io_log.py        # L3 LLM 入出力ログ
│   ├── l1/                      # L1 決定的レシピ/ルールエンジン
│   ├── l2_bayesian/             # L2 ベイズ最適化（Ax/BoTorch）
│   ├── l3_orchestrator.py       # L3 LLM オーケストレータ
│   ├── hmi/                     # 承認フロー・ダッシュボード API
│   ├── audit/                   # ALCOA-lite 監査ログ・event_store
│   └── gateway/                 # MQTT/LADS/SiLA2 gateway
├── sim/plant_model/             # 文献ベースのデジタルツイン
│   └── __init__.py              # Manstein 2021 モデルの scipy 実装
├── infra/                       # 通信・接続設定例
│   └── virtual_edge/            # MQTT 経由の結線確認プロファイル
└── docs/design/                 # 設計文書
    ├── adr/0001-control-architecture.md
    ├── kg_to_auto_cell.md
    ├── roadmap.md
    └── regulatory_technical_controls.md
```

### 4.1 細胞培養プラグイン

`plugins/cell_culture/` は、`physical-ai-core` という共通フレームワーク上に構築された領域特化プラグインである。細胞培養特有のルールをコード化し、フレームワークに以下を提供する。

- **environment model**: 培養液の状態を表すデータ構造（VCD、グルコース、乳酸、DO、pH、凝集体径など）。
- **channel routing**: センサ名から内部状態へのマッピングルール。
- **event detection**: 「乳酸高」「DO 低」「凝集体径アウト」「VCD 目標到達」などのイベント判定。
- **tool schemas**: システムが実行可能な操作の定義と引数。
- **validation rules**: 各操作の安全範囲（灌流率の変化速度、撹拌 rpm の上限、継代時の Y-27632 同時添加の強制など）。
- **aggregate imaging**: at-line 明視野/位相差画像から凝集体径、大径割合、円形度、アスペクト比を算出。
- **confidence scoring**: GP/PLS/DL モデルの不確実性を [0,1] スコアに正規化し、低信頼度時に人間へエスカレーション。
- **LLM prompts**: 生物学的制約を LLM に伝達するシステムプロンプト。

### 4.2 デジタルツイン

`sim/plant_model/` は、実機を使用せずに制御ロジックを検証するためのシミュレーション環境である。

- Manstein, Ullmann, Triebert & Zweigerdt 2021 の 6 項 Monod 型速度論モデルを scipy で再実装している。
- アクチュエータ入力に対するセンサ時系列出力を返すインターフェースにより、制御ループを閉じて検証可能である。
- 検証目標：7 日間で 35×10⁶ cells/mL、DO 40%→10%、pH 7.1。
- 将来的に COBRApy + GEM または商用 co-sim ツールへバックエンドを差し替え可能なインターフェースを維持する。

本ツインは、制御ループの決定性検証（同一入力に対する同一出力）を CI で実施することにより、信頼性を高める役割を担う。

### 4.3 L2 / L3 層（骨格）

`src/auto_cell/l2_bayesian/` と `src/auto_cell/l3_orchestrator.py` は、Phase 1 で骨格として導入された最適化・判断支援層である。

- **L2 ベイズ最適化**: Ax/BoTorch を使い、seeding density、初期グルコース、灌流 ramp profile、最大灌流率、撹拌 rpm、DO transition、Y-27632 濃度などを run 間で最適化する。固定シード再現性テストを CI で実施する。
- **L3 LLM オーケストレータ**: 承認仲介、曖昧な知覚、新規例外、研究者対話などの非クリティカル用途で起動する。プロンプトは Jinja2 でバージョニングし、SHA256 ハッシュで追跡する。入出力は不変ログに保存する。

L2/L3 の出力は必ず L1 の安全包絡線・承認フローを通過する。クリティカルな高速制御には使用しない。

### 4.4 通信とデバイス接続

`infra/` は、バイオリアクターや周辺機器との接続を担う通信層である。

| 標準 | 用途 |
|---|---|
| **MQTT** | システムの「ブレイン」と中継ゲートウェイ間の主要通信。軽量でリアルタイム性に優れる。 |
| **OPC-UA + LADS** | バイオリアクター本体のプロセス制御。センサ、アクチュエータ、プログラムを統一的にモデル化する。 |
| **SiLA2** | サンプリングロボ、分注装置、at-line 分析器などのラボ機器接続。 |

システムは MQTT-native で動作し、ゲートウェイが OPC-UA/LADS および SiLA2 とのプロトコル変換を行う。これにより、装置ベンダーが異なっても、ブレイン側のロジックを変更することなく接続可能である。

## 5. AI/ML の信頼性と規制対応

医療・製薬分野において AI を使用する場合、判断根拠の説明、再現性、および人間による介入可能性が要求される。

### 5.1 安全包絡線

LLM または最適化エンジンからの指示であっても、以下の決定的な検証ルールを通過しなければ実行されない。

- 各操作の絶対上限・下限
- 変化速度の上限（例：灌流率の急激な変更の禁止）
- 禁止状態遷移（例：凝集体解離時の Y-27632 同時添加の強制）
- 汚染疑い時の即時停止

これにより、AI の予期しない出力が装置を動かすリスクを低減する。

### 5.2 説明可能性と信頼度

- **XAI（Explainable AI）**: 判断に寄与した重要な特徴量を提示する。
- **信頼度スコア**: 判断の確からしさを数値化し、低い場合は人間へのエスカレーションを行う。
- **プロンプトバージョニング**: LLM への指示文の変更履歴を管理し、再現性を確保する。

### 5.3 監査証跡

すべてのセンサデータおよび操作は、以下の情報とともに不変に記録される。

- 実行者、時刻、操作内容、理由（who/when/what/why）
- 操作前後の状態
- 承認者と承認時刻
- AI 判断の根拠（該当する場合）

これは電子バッチ記録（EBR）、ALCOA+、および FDA 21 CFR Part 11 への対応を見据えた設計である。

### 5.4 データ完全性と再現性

- 訓練データ、検証データ、運用データの分離
- モデルカードによる学習データと性能限界の文書化
- 固定シードによる再現性テスト
- ドリフト監視によるモデル性能劣化の検知

## 6. 開発・実行方法

### 6.1 環境構築

```bash
# Python 3.11 以上が必要
# 基本開発環境
uv sync --extra dev

# L2 BO + L3 LLM + 凝集体画像解析を含む完全環境
uv sync --extra bo --extra llm --extra imaging --extra dev
```

### 6.2 テスト実行

```bash
uv run pytest
```

### 6.3 依存関係

本プロジェクトは `physical-ai-core` という共通フレームワーク上に構築されている。`uv sync` 実行時には、`../physical-ai-core` を editable モードで参照する。

重いライブラリ（Ax/BoTorch/torch、openai、cellpose 等）は optional dependencies として分離している。`pyproject.toml` の `[project.optional-dependencies]` を参照。

現時点では Phase 1（R&D 段階）であり、実装置への接続は今後のフェーズで実施する予定である。

## 7. 用語解説

| 用語 | 意味 |
|------|------|
| **LLM** | Large Language Model。大規模言語モデル。自然言語による状況説明や判断に使用する。 |
| **ReAct** | Reasoning + Acting。AI が「推論→行動→観察」を繰り返す方式。 |
| **Human-in-the-loop / Human-on-the-loop** | 人間を制御ループに組み込む設計。重要な判断で AI が提案し、人間が承認または監視する。 |
| **WorldModel** | システム内部に構築された、培養プロセスの現在状態を表すモデル。 |
| **Digital Twin** | 実際の装置やプロセスをコンピュータ上に再現したモデル。本プロジェクトでは `sim/plant_model` が該当する。 |
| **OPC-UA** | 産業用機器間の通信規格。セキュアかつ型付きのデータ交換が可能である。 |
| **LADS** | OPC-UA 上のバイオリアクター・分析機器向け情報モデル。 |
| **SiLA2** | ラボ機器向け通信規格。HTTP/2 + protobuf をベースとする。 |
| **MQTT** | 軽量なメッセージ通信プロトコル。IoT やエッジデバイスで広く使用される。 |
| **ALCOA+** | 製薬業界のデータ完全性原則。Attributable, Legible, Contemporaneous, Original, Accurate など。 |
| **GxP / Part 11** | 医薬品製造・品質管理の規制要件。電子記録・電子署名に関する FDA 規則。 |
| **CSV/CSA** | Computerized System Validation / Computer Software Assurance。製薬用ソフトウェアの検証アプローチ。 |
